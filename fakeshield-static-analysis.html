E-11 BlockchainRecord Evidence & Audit An immutable Hyperledger Fabric transaction recording a case lifecycle event (EVIDENCE_LOCKED, TAKEDOWN_SENT, PLATFORM_RESPONSE, ESCALATED). txId, eventType, evidenceHash, blockNumber, channelName, timestamp
E-12 TakedownRequest Communication A formal delete/suspend request sent by the agency to a social media platform. Tracked for SLA compliance; linked to blockchain record. requestNumber, requestedAction, status, sentAt, slaDeadline, reminderCount
E-13 Notification Communication System-generated message to Citizens (status updates), Platforms (SLA reminders), or Officers (alerts). Sent via email/SMS/in-app. type, channel, recipient, message, sentAt, isDelivered
E-14 LegalReport Communication A court-admissible PDF dossier generated for escalated cases. Contains the full case timeline, evidence hashes, blockchain TX IDs and digital signature. reportNumber, digitalSignature, blockchainTxId, applicableSections[ ]

R-01 Citizen submits Report 1 — N Association One citizen can submit many reports; each report is from exactly one citizen.
R-02 Report generates Case 1 — 1 Association Each verified report creates exactly one Case. A Case may also originate from a proactive scan (no Report).
R-03 Officer manages Case M — N Association An officer can manage many cases; a case can be reassigned to different officers over its lifecycle. Junction: CaseAssignment.
R-04 Officer assignedTo Role N — 1 Association Many officers share a role; each officer has exactly one role at a time.
R-05 Case investigates SocialAccount N — 1 Association Multiple cases can target the same account (re-opened, re-reported). Each case targets exactly one account.
R-06 SocialAccount hostedOn Platform N — 1 Association A platform hosts many accounts; each account belongs to exactly one platform.
R-07 Platform governs SLAConfig 1 — 1 Composition ◆ Each platform has exactly one SLA configuration. SLAConfig cannot exist without its Platform (strong lifecycle dependency).
R-08 Case hasResult ScanResult 1 — 1 Composition ◆ Each case has exactly one scan result. ScanResult is owned by its Case and deleted when the case is purged.
R-09 Case contains Evidence 1 — N Composition ◆ A case must have at least one evidence item; multiple evidence types (screenshot, metadata, NLP report, graph data) can exist per case.
R-10 Case generates TakedownRequest 1 — N Composition ◆ A case can produce multiple takedown requests (e.g., re-sent after SLA breach). Each request belongs to one case.
R-11 TakedownRequest sentTo Platform N — 1 Association Each takedown request is directed at one platform; a platform receives many takedown requests across all cases.
R-12 Evidence lockedIn BlockchainRecord 1 — 1 Association Each confirmed evidence bundle produces exactly one blockchain transaction. The hash links the physical evidence to the immutable ledger entry.
R-13 Case triggers Notification 1 — N Aggregation ◇ A case generates many notifications (acknowledgment, status update, SLA reminder, escalation alert). Notifications can survive case archival.
R-14 Case produces LegalReport 1 — 0..1 Association A legal report is optional — generated only for escalated cases. At most one legal report per case at any time.
R-15 LegalReport verifiedBy BlockchainRecord 1 — 1 Association Every legal report embeds the blockchain TX ID of the evidence lock transaction to prove tamper-proof integrity in court.
